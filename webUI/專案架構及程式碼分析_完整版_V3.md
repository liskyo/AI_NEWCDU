# Smart Warehouse Dynamic Management System / Guchii CDU Control
## 專案架構及程式碼分析_完整版_V3

本文件提供對「Guchii CDU Control (冷卻分配單元) 控制系統」專案架構與程式碼的完整深度分析。
這是一個結合前端 Vue 3 SPA (Single Page Application) 與後端 Python Flask，並透過 Modbus TCP 與底層 PLC 設備進行通訊的工業控制系統。

---
 
## 1. 系統整架構圖 (System Architecture)

```mermaid
graph TD
    User([User Browser]) <-->|HTTP / WebSocket / Axios| Vue[Frontend: Vue 3 + Vite + TailwindCSS]
    Vue <-->|REST API (JSON)| Flask[Backend: Python Flask Server]
    
    subgraph "Backend Services (Python)"
        Flask <--> GlobalState[(Global Memory State\nsensorData\nctr_data)]
        GlobalState <--> ModbusThread[Background Thread: read_modbus_data]
        Flask <--> JsonStore[(Disk JSON Storage\nLogs/Parameters)]
    end
    
    ModbusThread <-->|Modbus TCP| PLC[Industrial PLC / Equipment]
    Flask <-->|REST API| SNMP[SNMP RestAPI Service]
```

### 1.1 核心技術堆疊 (Technology Stack)
* **Frontend (前端)**
  * **Framework**: Vue 3 (Composition API)
  * **Build Tool**: Vite
  * **Styling**: TailwindCSS, Heroicons (SVG icons)
  * **Routing**: Vue Router
  * **HTTP Client**: Axios
  * **Charting**: (部分採用 HTML5 Canvas 或是手動繪製 SVG 折線圖如 Trends.vue)
* **Backend (後端)**
  * **Framework**: Python Flask
  * **CORS**: Flask-CORS (允許跨域請求)
  * **Auth**: Flask-Login, Werkzeug Security (目前因為在地端運行，許多路由已開放給前端直接存取)
  * **Hardware Comms**: PyModbus (透過 TCP 讀取/寫入 Holding Registers 與 Coils)
  * **Logging**: 內建 Python `logging` 模組 (分層為 system, error, operation)

## 2. 專案目錄結構分析 (Directory Structure)

專案根目錄為 `webUI/`，其下主要分為兩個大區塊：`frontend/` (前端原始碼) 與 `web/` (後端原始碼)。

### 2.1 控制層 (專案根目錄 Scripts)
* `webui_service.sh` / `.bat`: 系統啟動腳本。
* `requirements.txt`: Python 後端依賴套件清單。
* `generate_test_logs.py`: 獨立的測試日誌生成腳本。

### 2.2 前端目錄結構 (`webUI/frontend/`)
現代化的 Vue 3 專案結構。
* `package.json`: NPM 配置與依賴 (Axios, Vue, Tailwind 等)。
* `vite.config.js`: Vite 打包配置 (設定 proxy 把 `/api` 導向 Flask 5501 Port 以避開跨域問題)。
* `tailwind.config.js`: Tailwind CSS 主題與樣式設定。
* `index.html`: Web 應用程式進入點。
* **`src/`**
  * `main.js`: 應用程式初始化，掛載 Router 與根元件。
  * `App.vue`: **全域主佈局 (Root Component)**。
    * 負責呈現上方導航列 (Navbar)，包含：中英文多國語系切換 (i18n)、Kiosk Mode (展示/戰情室模式) 切換。
    * 透過 Router-View 掛載各分子頁面。
  * **`router/`**
    * `index.js`: Vue Router 設定。定義了包含 `/status`, `/control`, `/trends`, `/logs`, `/engineer-mode` 等核心路由。
  * **`components/` (核心頁面元件)**
    * `StatusEngineer.vue`: **狀態總覽 (Status)**。即時顯示冷卻水、廠務水、溫濕度等感測器數據的儀表板。
    * `Control.vue`: **設備控制 (Control)**。操作 PLC 控制閥門、幫浦的開關與頻率。
    * `Trends.vue`: **趨勢分析 (Trends)**。讀取歷史或即時數據繪製折線圖，支援「TEST MODE」全域展示狀態同步。
    * `Logs.vue`: **系統日誌 (Logs)**。呈現系統發生的事件紀錄 (Alarm/Warning/Info)，包含動態搜尋與過濾機制。
    * `EngineerMode.vue`: **工程模式 (Engineer)**。管理底層警報閾值 (Thresholds) 與設備初始參數設定。
    * `Settings.vue` / `Maintenance.vue` / `Backup.vue` / `Network.vue`: 各自對應系統的附屬設定與維護功能。

### 2.3 後端目錄結構 (`webUI/web/`)
Python Flask 伺服器與資料儲存層。
* **`app.py`**: **後端核心入口程式 (The Monolith Application)**。
  * 這個檔案非常龐大 (超過 8000 行)，包含了所有的 Flask API 路由 (Routes)、後台監控執行緒 (Background Threads)、與 PLC 的通訊邏輯 (ModbusTCP)，以及假資料的模擬邏輯 (DEMO_MODE)。
* `server.ini`: gunicorn/uwsgi 或其他伺服器的組態檔設定。
* `measure_data.json`: 系統斷電重啟時的參數暫存 (有時也做為快取用途)。
* `fw_info.json`: 韌體資訊。
* **`json/` (持久化資料儲存區)**
  * `signal_records.json`: 系統警報/資訊的一般日誌紀錄。
  * `downtime_signal_records.json`: 導致系統停機的重大警報紀錄 (Shutdown Errors)。
  * `system_data.json` / `system_setting.json`: 系統的基礎設定值。

## 3. 核心運行機制與資料流向 (Core Mechanisms)

### 3.1 前後端資料同步 (Data Synchronization)
本系統並未採用 WebSocket 推播，而是由前端 Vue 應用程式在各個元件的 `onMounted` 生命週期中，開啟 `setInterval` (通常為 1-5 秒) 週期性地向後端發送 HTTP GET 請求 (如 `/api/status`, `/get_data_control`)。

1. **Frontend**: `axios.get('/api/status')`
2. **Backend**: 攔截路由，將存在系統記憶體中的全域變數 `sensorData` (字典物件) 轉換成 JSON 回傳。
3. **Frontend**: Vue 將回傳的新數據賦值給 `ref` 或 `reactive` 變數，觸發畫面 DOM 響應式更新 (Re-render)。

### 3.2 後端硬體聯網層 (Modbus Polling Thread)
在 `app.py` 啟動時，會連帶啟動一條無窮迴圈的背景執行緒：`read_modbus_data()`。
1. 每隔一秒或更短的時間，透過 `ModbusTcpClient` 連線到 `modbus_host` (PLC 的 IP 位置)。
2. 使用 `client.read_holding_registers()` 及 `client.read_coils()` 向 PLC 索取溫濕度、壓力、硬體狀態。
3. 讀取完畢後，透過 `BinaryPayloadDecoder` 進行字節序轉換 (Endianness)，解譯成實際數值，然後**更新到 Flask 全域變數 `sensorData` 與 `ctr_data` 中**。
4. **警報判斷**: 執行緒內也會同時依據設定好的上下限閾值，檢查當前數值是否超標。若超標超過特定次數，會呼叫 `record_signal_on()` 把 Error/Warning 寫入 `signal_records.json` 檔案中，供 `Logs.vue` 讀取。

### 3.3 TEST MODE / 展示模式 (Demo Simulation Engine)
本系統提供一個強大的無硬體展示功能，名為 **TEST MODE (全域展示模式)**，其架構如下：

* **開啟機制**: 
  在前端任何具有右上角「TEST MODE」切換按鈕的頁面 (例如 Status, Trends, Logs)，點擊後會發送 `POST /api/toggle_demo_mode` 給後端，改變全域變數 `DEMO_MODE` 的布林值。
  
* **後端攔截 (Bypass Strategy)**:
  1. 一旦 `DEMO_MODE = True`，背景執行緒 `read_modbus_data()` 會**完全略過**對真實 PLC 的連線 (防止系統因為找不到硬體而產生 504 Timeout 與 M351 斷線警報)。
  2. 系統改為呼叫 `generate_demo_sensor_data()`。此函式會利用 `math.sin()` (三角函數) 配合當前時間戳，完美模擬出具有平滑起伏變化的溫度 (`tempSim`)、流量 (`flowSim`) 與壓力數值。

* **異常動態注入 (Dynamic Mock Events)**:
  當處於展示模式時，為了讓 Logs 分頁有生動的展示內容，系統每 15 秒會透過 Python `random` 模組，隨機產生一個系統警報 (例：`M2_Pump_Fail` 或 `M3_CoolantTemp_High`)，呼叫 `record_signal_on` 寫入日誌清單，並且在再過 15 秒後呼叫 `record_signal_off` (標示 Restore)，完美重現警報發生到解除的系統行為。

* **UI 全域同步**:
  不論從哪個分頁開啟，前端皆會在載入時透過 `GET /api/demo_mode_status` 與後端對齊狀態。所有對 `Trends` (趨勢圖) 的資料請求也都會直接拿取這份完美的模擬記憶體資料，達到毫無破綻的動態展示效果。

## 4. 進階前端功能與雙語機制 (Advanced Frontend Features)

### 4.1 UI 響應式與風格 (Styling)
* 使用 **Tailwind CSS V4** 取代傳統手寫 CSS。高度利用 Utility-classes (如 `flex`, `items-center`, `justify-between`) 快速建構出具備 Cyberpunk / 工控質感 (深色底 `bg-gray-900` + 青色文字 `text-cyan-400`) 的介面。
* 所有圖示皆採用 `@heroicons/vue`，確保在各種解析度 (包含 Kiosk 戰情室大螢幕) 下皆能保持銳利的向量渲染。

### 4.2 Kiosk Mode (戰情室展演模式)
* 定義在 `App.vue` 中的 `toggleKioskMode()`。
* 當啟動時，系統會呼叫瀏覽器原生 `requestFullscreen()` 進入全螢幕。
* 隱藏最上方的導航列 (Navbar)，並啟動 `setInterval`，讓畫面每 30 秒自動在 `/status` 與 `/trends` 之間進行切換，達到無人看管自動輪播的展演效果。

### 4.3 輕量級雙語切換 (Lightweight i18n)
* 為求效能不載入龐大的 `vue-i18n` 套件，本專案於 `App.vue` 中實作了精巧的響應式 (Reactive) 字典對應機制。
* **狀態保存**: 透過瀏覽器內建的 `localStorage.getItem('preferredLang')` 記憶使用者的習慣語言，保證 F5 重新整理後語言不會跑版。
* **即時切換**: 定義了 `currentLang` Ref 變數與 `t(key)` 函式，只需變更 Ref 的值，Vue 立即更新 DOM 提供中 (繁體) 英無縫切換。所有選用的繁體中文詞彙皆採用道地工業控制術語 (如: 狀態總覽、維護保養、趨勢分析)。

## 5. 總結與維護建議 (Conclusion & Maintenance)

本專案將複雜的工業 Modbus 通訊與現代 Web UI 深度結合。目前的展示引擎 (Demo Engine) 機制相當成熟，不僅能保證無硬體時的穩定展示，更能自動模擬真實的系統行為。

### 未來優化建議 (Future Recommendations)
1. **後端模組化 (Refactor `app.py`)**: 
   `app.py` 目前行數龐大，將所有 Routes、MQTT/SNMP 邏輯、及 Modbus 控制集中在單一檔案。建議未來可以採用 Flask Blueprints 進行拆分 (例如切分出 `routes_api.py`, `modbus_client.py`, `demo_simulator.py`)，將有助於程式碼可讀性及測試。
2. **WebSocket 導入**:
   感測器數據目前採用前端 AJAX 高頻率 Polling (每秒甚至更短) 向後端索取，這會造成前端頻繁發送 HTTP Header 導致無謂的頻寬損耗。導入 Flask-SocketIO 後端主動推播 (Push) 改變的參數，可大幅降低系統 IO 與網路負載。
