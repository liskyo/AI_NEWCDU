# 專案架構及程式碼分析

## 1. 專案概述 (Project Overview)
這是一個用於 **CDU (Coolant Distribution Unit, 冷卻液分配單元)** 的工業控制系統軟體專案。該系統負責監控與控制冷卻液的溫度、壓力、流量、水質 (pH, 電導率, 濁度) 等關鍵參數，並管理泵浦 (Pumps)、閥門 (Valves) 與過濾器 (Filters) 的運作狀態。

系統採用分層架構，主要包含 **PLC 控制層**、**RestAPI 介面層** 以及 **WebUI 前端展示層**。

## 2. 專案目錄結構 (Directory Structure)

專案根目錄包含了多個模組化資料夾與部署腳本：

- **`PLC/`**: 核心控制邏輯，包含與底層硬體 (Modbus 設備) 通訊及自動化控制的 Python 程式。
- **`RestAPI/`**: 提供 RESTful API 介面，供前端或外部系統獲取即時數據與發送控制指令。
- **`webUI/`**: Web 前端介面相關檔案，用於操作員監控與管理。
- **`snmp/`**: 可能涉及透過 SNMP 協定進行網路設備監控或被監控。
- **`modbus_proxy/`**: Modbus 代理服務，可能用於多個客戶端同時存取 Modbus 設備。
- **`logs/`**: 系統運行日誌存放處。
- **`scripts (*.sh)`**: 
    - `deploy.sh`, `network.sh`, `restart.sh`: 系統部署、網路設定與服務重啟腳本。

## 3. 核心模組程式碼分析 (Code Analysis)

### 3.1 PLC 控制模組 (`PLC/plc.py`)
這是系統的「大腦」，負責實際的邏輯運算與硬體互動。

- **通訊協定**: 使用 `pymodbus` (TCP & Serial) 與下層設備通訊。
- **主要功能**:
    - **數據採集 (Data Acquisition)**: 週期性讀取感測器數值（溫度、壓力、流量、濕度、電力參數等）。
    - **邏輯控制**: 包含泵浦的啟停控制、轉速調整 (Inverter speed)、閥門開關 (EV1-EV4) 以及過濾器狀態管理。
    - **狀態監控**: 維護大量的狀態變數（如 `f1_data`, `p1_data`），用於偵測異常與警報觸發。
    - **設備輪替 (Redundancy)**: 程式碼中包含 `pump_swap_time` 邏輯，顯示系統支援泵浦輪替運轉以延長壽命。
    - **日誌記錄**: 使用 `ConcurrentTimedRotatingFileHandler` 進行日誌輪替，確保長時間運行的紀錄穩定性。
- **關鍵變數**:
    - `bit_output_regs`, `word_regs`: 定義了寫入 PLC/設備的控制暫存器。
    - `sensor_raw`, `ad_sensor_value`: 儲存原始數據與轉換後的工程數值。
    - `inspection_data`: 用於存儲設備巡檢或自檢的結果與進度。

### 3.2 RestAPI 模組 (`RestAPI/app.py`, `RestAPI/rest_api_app.py`)
這是系統的「介面」，連接前端與底層數據。

- **框架**: 基於 Python `Flask` 框架，並使用 `flask_restx` 進行 API 文檔化與管理。
- **API 結構**: 定義了 `api/v1` 命名空間。
- **數據模型**:
    - **`sensor_value_data`**: 定義了所有感測器的詳細屬性（名稱、數值、單位、狀態、警報閾值）。這是 API 回傳給前端的主要數據結構。
    - **`messages`**: 集中管理所有的 Warning (M1xx), Alert (M2xx), Error (M3xx) 代碼與描述，便於前端顯示多語言或標準化錯誤訊息。
- **功能**:
    - 提供 API 讓 WebUI 獲取即時感測器數值 (`data` 字典)。
    - 接收前端設定參數（如溫度設定、壓力設定、泵浦轉速設定）。
    - 處理警報與事件日誌的查詢。

### 3.3 WebUI 模組 (`webUI/web/`)
- 包含前端網頁資源 (`templates/`, `static/`) 與相關設定 (`json/`)。
- `json/` 目錄下可能存放前端的設定檔或暫存狀態。

## 4. 數據流向 (Data Flow)

1. **感測層**：實體感測器 (溫度、壓力等) 與執行器 (泵浦、閥門)。
2. **採集層 (`PLC/plc.py`)**：透過 Modbus 協議輪詢實體設備，將原始數據轉換為工程數值，並執行自動化控制邏輯 (如過壓保護、自動切換)。
3. **數據層**：`PLC` 模組可能更新共享記憶體、檔案或 Modbus 暫存器（視具體實作而定，`RestAPI` 似乎也獨立連線 Modbus 或讀取共享資料）。
4. **服務層 (`RestAPI`)**：將數據封裝成 JSON 格式，透過 HTTP 提供給前端。
5. **展示層 (`WebUI`)**：操作員透過瀏覽器查看即時數據、趨勢圖與警報，並下達控制指令。

## 5. 技術棧 (Tech Stack)
- **語言**: Python 3, Shell Script
- **Web 框架**: Flask, Flask-RESTX
- **通訊協定**: Modbus TCP/RTU
- **伺服器**: Gunicorn (見 `RestAPI/gunicorn_config.py`), Nginx (見 `nginx/` 目錄)
- **依賴管理**: `requirements.txt` (包含 `pymodbus`, `flask`, `flask_limiter` 等)

## 6. 結論與建議
- 專案結構清晰，職責分離明確（控制 vs 介面）。
- **`PLC/plc.py`** 邏輯較為複雜，單一檔案行數較多，未來維護時建議可依功能（如「泵浦控制」、「感測器讀取」、「警報處理」）拆分為多個模組。
- **錯誤處理** 相當完善，定義了詳盡的錯誤代碼表 (`RestAPI/app.py` 中的 `messages`)。
- 建議在開發新功能時，與現有的 `sensor_value_data` 結構保持一致，以利於前端統一處理。
